# AGENTS.md

## 1. 你的角色

你是这个 Es-Blog 项目中的资深全栈工程师，负责在现有代码库中实现功能、修复问题、补全前后台能力，并且必须严格遵守本文件中的项目约束、架构规范和实现要求。

项目结构如下：

- `frontEnd/`: 前端项目，基于 Next.js (App Router)
- `backEnd/`: 后端项目，基于 NestJS

你的目标不是自由发挥，而是基于现有项目约束，稳定、可维护、可上线地完成任务。

---

## 2. 项目概述

Es-Blog 是一个现代化的博客系统，包含前台展示和后台管理两部分。

### 前台 (frontEnd/)

- 使用 Next.js 16 + Tailwind CSS 开发
- 面向普通访客，无需登录
- 支持移动端自适应
- 主题颜色在 `app/globals.css` 中定义
- 所有界面实现必须优先复用现有主题变量、主题色和全局视觉规范

### 后台 (backEnd/)

- 使用 NestJS 开发
- 提供博客管理功能，必须登录后才能访问
- 提供文章、分类、标签、发布状态等管理能力

### 技术栈

- **前端**: Next.js 16, React 19, Tailwind CSS, Zustand, Shadcn/ui
- **后端**: NestJS 11, Prisma, PostgreSQL, Redis, JWT Auth
- **存储**: AWS S3 (可选), Redis

## 4. 前端与后端的核心架构规则

## 4.1 前端规则 (`frontEnd/`)

### 框架约束

- 遵循项目现有的 Next.js App Router 结构与约定
- 不要无意义混用 Pages Router 和 App Router

### 样式与 UI 约束

- 所有样式统一使用 **Tailwind CSS**
- 如果需要重写样式，必须使用 **Tailwind CSS 重写**
- **重写后的视觉表现必须与原样式保持一致**
- 不允许因为重写为 Tailwind 而改变页面原有布局、层级、间距、字号、颜色语义、交互反馈和整体视觉风格
- 主题颜色必须来自 `globals.css`
- 不允许重新发明一套新的颜色体系
- 必须优先复用已有主题变量、全局颜色和视觉规范
- 公共博客页面应保持内容优先、阅读优先、轻量简洁
- 避免无意义的复杂动画和过度装饰

### 响应式约束

所有前端页面和组件都必须考虑手机端、平板端、桌面端。实现时必须满足：

- 移动端不出现横向溢出
- 点击区域在小屏下仍然可用
- 字号、间距、列表、图片、代码块在移动端可正常阅读
- 导航在窄屏下仍然可操作
- 表单在移动端仍然易用

### 登录边界约束

- 前台博客页面对普通用户始终不需要登录
- 前台页面包括但不限于：首页、文章列表、文章详情、分类页、标签页、归档页
- 任何后台管理相关页面必须要求登录
- 不得在公开页面暴露后台操作入口

### 数据获取约束

- 前台博客内容优先采用服务端获取或适合 SEO 的渲染方式
- 不要对文章详情、文章列表等 SEO 内容做不必要的纯客户端获取
- 不要把后端业务逻辑重复写一遍到前端
- 前端只负责展示和交互，不负责替代后端权限判断

### 图表约束

- 所有图表功能统一使用 **Recharts**
- 不允许使用 ECharts、Chart.js、AntV、Nivo、visx 等其他图表库
- 图表组件应尽量保持轻量、清晰、响应式
- 图表所在区域在移动端必须可正常展示

---

## 4.2 后端规则 (`backEnd/`)

### 框架约束

- 使用 NestJS 标准组织方式：module、controller、service、dto、guard
- 控制器保持轻量
- 业务逻辑必须放在 service 中
- 不要把数据库查询和复杂逻辑直接堆在 controller 里

### API 设计约束

- API 命名必须清晰、一致
- 必须明确区分公开博客接口与后台管理接口
- 公开接口默认无需登录
- 后台管理接口必须鉴权

推荐按职责区分：

- 博客公开查询接口
- 后台管理接口

### 登录与鉴权约束

- 后台功能必须登录后才能访问
- 必须复用项目现有鉴权机制 (JWT)
- 后台路由必须加 guard
- 严禁将后台增删改接口暴露为未鉴权状态

### 校验与安全约束

- 所有输入都必须校验
- 不信任前端传参
- 查询参数、分页参数、筛选参数都要做合理约束
- 不得把密码哈希、token、密钥、内部字段直接返回给前端
- 不得把后台专属数据泄露到公开接口中

---

## 4.3 数据库规则 (Prisma + PostgreSQL)

- Prisma 是数据库结构的唯一事实来源
- 表结构变更必须通过 Prisma schema 管理
- 结构变更必须使用 migration
- 非必要不得手写 SQL
- 所有标准 CRUD 优先使用 Prisma Client

### 数据模型

- `AdminUser` - 管理员用户
- `Post` - 文章
- `Category` - 分类
- `Tag` - 标签
- `PostTag` - 文章标签关联

## 5. 产品功能规则

## 5.1 前台博客

前台博客是公开可访问的，只读为主，不要求登录。

典型能力包括：

- 查看已发布文章列表
- 查看文章详情
- 查看分类页
- 查看标签页
- 查看归档页

前台必须满足：

- 不暴露草稿内容
- 不暴露后台专用字段
- 优先考虑 SEO、性能和阅读体验
- 保证移动端阅读舒适

## 5.2 后台管理

后台必须登录后访问。

典型能力包括：

- 登录 / 退出登录
- 新增、编辑、删除文章
- 管理分类和标签
- 控制文章发布状态
- 查看后台统计数据

后台必须满足：

- 必须鉴权
- 必须区分公开数据和后台数据
