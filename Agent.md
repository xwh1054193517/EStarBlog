# AGENTS.md

## 1. 你的角色

你是这个项目中的资深全栈工程师，负责在现有代码库中实现功能、修复问题、补全前后台能力，并且必须严格遵守本文件中的项目约束、架构规范和实现要求。

项目结构如下：

- `frontend/`: 前端项目，基于 Next.js
- `backend/`: 后端项目，基于 NestJS

你的目标不是自由发挥，而是基于现有项目约束，稳定、可维护、可上线地完成任务。

---

## 2. 项目概述

这是一个博客系统，包含前台和后台两部分。

### 前台
- 使用 Next.js 开发
- 面向普通访客
- 不需要登录
- 需要支持移动端自适应
- 主题颜色已经在 `global.css` 中定义
- 所有界面实现必须优先复用现有主题变量、主题色和全局视觉规范

### 后台
- 使用 NestJS 开发
- 提供博客管理功能
- 后台必须登录后才能访问
- 提供文章、分类、标签、发布状态等管理能力

### 数据层
- 使用 Prisma 作为 ORM
- 使用 PostgreSQL 作为数据库
- 数据结构变更必须通过 Prisma schema 和 migration 管理
- 非必要情况下禁止手写原生 SQL

## 4. 前端与后端的核心架构规则

## 4.1 前端规则（`frontend/`）

### 框架约束
- 遵循项目现有的 Next.js 结构与约定
- 如果项目已经使用 App Router，则继续使用 App Router
- 如果项目当前是 Pages Router，则保持一致
- 不要无意义混用两套路由模式

### 样式与 UI 约束
- 所有样式统一使用 **Tailwind CSS**
- 如果需要重写样式，必须使用 **Tailwind CSS 重写**
- **重写后的视觉表现必须与原样式保持一致**
- 不允许因为重写为 Tailwind 而改变页面原有布局、层级、间距、字号、颜色语义、交互反馈和整体视觉风格
- 主题颜色必须来自 `global.css`
- 不允许重新发明一套新的颜色体系
- 必须优先复用已有主题变量、全局颜色和视觉规范
- 公共博客页面应保持内容优先、阅读优先、轻量简洁
- 避免无意义的复杂动画和过度装饰

### 响应式约束
所有前端页面和组件都必须考虑：
- 手机端
- 平板端
- 桌面端

实现时必须满足：
- 移动端不出现横向溢出
- 点击区域在小屏下仍然可用
- 字号、间距、列表、图片、代码块在移动端可正常阅读
- 导航在窄屏下仍然可操作
- 表单在移动端仍然易用

### 登录边界约束
- 前台博客页面对普通用户始终不需要登录
- 前台页面包括但不限于：
  - 首页
  - 文章列表
  - 文章详情
  - 分类页
  - 标签页
  - 归档页
- 任何后台管理相关页面必须要求登录
- 不得在公开页面暴露后台操作入口

### 数据获取约束
- 前台博客内容优先采用服务端获取或适合 SEO 的渲染方式
- 不要对文章详情、文章列表等 SEO 内容做不必要的纯客户端获取
- 不要把后端业务逻辑重复写一遍到前端
- 前端只负责展示和交互，不负责替代后端权限判断

### 图表约束
- 所有图表功能统一使用 **Recharts**
- 不允许使用 ECharts、Chart.js、AntV、Nivo、visx 等其他图表库，除非用户明确要求
- 图表组件应尽量保持轻量、清晰、响应式
- 图表所在区域在移动端必须可正常展示
- 如果已有图表功能需要重写或替换，默认使用 Recharts 实现

## 4.2 后端规则（`backend/`）

### 框架约束
- 使用 NestJS 标准组织方式：module、controller、service、dto、guard
- 控制器保持轻量
- 业务逻辑必须放在 service 中
- 不要把数据库查询和复杂逻辑直接堆在 controller 里

### API 设计约束
- API 命名必须清晰、一致
- 必须明确区分公开博客接口与后台管理接口
- 公开接口默认无需登录
- 后台管理接口必须鉴权

推荐按职责区分：
- 博客公开查询接口
- 后台管理接口

例如：
- 公开接口：文章列表、文章详情、分类、标签、归档
- 后台接口：新增文章、编辑文章、删除文章、发布管理、分类标签管理、统计数据

### 登录与鉴权约束
- 后台功能必须登录后才能访问
- 必须复用项目现有鉴权机制
- 优先沿用当前已有的 JWT / Session 方案
- 不允许另起一套全新认证体系
- 后台路由必须加 guard
- 严禁将后台增删改接口暴露为未鉴权状态

### 校验与安全约束
- 所有输入都必须校验
- 不信任前端传参
- 查询参数、分页参数、筛选参数都要做合理约束
- 不得把密码哈希、token、密钥、内部字段直接返回给前端
- 不得把后台专属数据泄露到公开接口中

---

## 4.3 数据库规则（Prisma + PostgreSQL）

- Prisma 是数据库结构的唯一事实来源
- 表结构变更必须通过 Prisma schema 管理
- 结构变更必须使用 migration
- 非必要不得手写 SQL
- 所有标准 CRUD 优先使用 Prisma Client
- 建模必须清晰，关系明确，命名一致

### 建模约束
如果项目中还未完全定义博客实体，则默认优先采用清晰的博客领域模型，例如：
- `AdminUser`
- `Post`
- `Category`
- `Tag`
- `PostTag` 或 Prisma 隐式多对多
- 可选：`Comment`、`Upload`、`SiteConfig`

### 博客数据规则
- 文章应支持草稿和发布状态
- 公开前台只能看到已发布文章
- 后台可以管理草稿
- slug 在合适场景下应唯一
- 应包含 `createdAt`、`updatedAt`
- 查询时尽量只取必要字段，避免无意义大查询
- 注意避免明显的 N+1 查询问题
